# Tri9T AI Engineering Engine: Compliance & Document Intelligence Platform

An audit-ready backend parsing and version-management system built using **FastAPI**, **SQLAlchemy + SQLite**, and **Pydantic V2** structured validation models. This platform converts raw, unstructured medical device PDF documentation into version-aligned object trees, tracks text mutations across manual versions ($v1 \rightarrow v2$), and manages resilient LLM QA test case generation with built-in staleness detection.

---

⚡ Setup & Installation Runbook
1. Initialize Your Virtual Environment
Open your PowerShell terminal inside the project directory and execute:

PowerShell
python -m venv venv
.\venv\Scripts\Activate.ps1
2. Install Package Dependencies
Ensure your environment runs the correct module versions:

PowerShell
pip install -r requirements.txt
3. Configure Runtime Environment Variables
Create a .env file in the root folder to house your active authentication tokens safely:

Plaintext
GROQ_API_KEY=your_free_tier_groq_api_key_here
DATABASE_URL=sqlite:///./tri9t_compliance.db
4. Execute Unit Tests
Run the automated test suite verifying your parsing logic and data handlers:

PowerShell
pytest
🚀 How to Execute the End-to-End Ingestion & Staleness Flow
Run the local development server to initialize the FastAPI app layer:

PowerShell
uvicorn app.main:app --reload
Open up your browser and navigate to http://127.0.0.1:8000/docs to access the interactive Swagger UI interface.

Step 1: Ingest Manual Version 1
Trigger the ingestion pipeline for the base version of the documentation by issuing a POST request to the /api/ingest-pipeline endpoint using these query values:

pdf_path: data/ct200_manual_v1.pdf

version: 1

Step 2: Generate a Pinned User Selection Snapshot
Bundle the critical safety parameters you wish to run quality control test matrices against by hitting POST /api/selections:

JSON
{
  "id": "sel_blood_pressure_rules",
  "name": "Cuff Overpressure Pinned Matrix",
  "node_ids": ["node_3_2_cuff_inflation_sequence_xxxx", "node_4_1_overpressure_protection_xxxx"],
  "version": 1
}
(Note: Replace the node_ids arrays with the exact logical IDs generated by the engine during Step 1).

Step 3: Trigger Resilient LLM QA Case Generation
Execute a POST request to /api/selections/sel_blood_pressure_rules/generate to run the reconstructed text blocks through the Pydantic self-correcting validation loop and generate structured QA plans.

Step 4: Ingest the Updated Manual Version 2
Simulate a regulatory updates cycle by re-triggering POST /api/ingest-pipeline with these adjusted parameters:

pdf_path: data/ct200_manual_v2.pdf

version: 2

Step 5: Audit Live Staleness and Text Decay Metrics
Issue a GET request targeting /api/selections/sel_blood_pressure_rules/testcases.

The retrieval system will cross-reference the hash signatures snapshot on your version 1 selection bounds against the incoming updates from version 2. It will output a live staleness_impact_report identifying exactly which test suites require revision because the primary source parameters mutated inside the new manual version.
